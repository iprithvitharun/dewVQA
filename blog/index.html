Building Dew VQA: When We Stopped Matching Elements and Let AI See

A Chrome extension for automated visual QA testing. What we built, what broke, and why we ended up sending screenshots to Claude instead of writing a matching algorithm.

Prithvi Tharun
UX Engineer
Mamtha
QA Lead — originated the idea
Chapter 1

The Problem

Visual QA testing at Freshworks is largely manual. A designer opens Figma on one side, the dev build on the other, and compares them pixel by pixel. Is the heading 24px or 28px? Is that border color #eaeef2 or #e0e0e0? Is the button text "Sign In" or "Sign in"?

This takes hours, catches maybe 60% of issues, and nobody enjoys doing it.

Mamtha, our QA lead, had the first insight: what if we could automate this? Not pixel-perfect screenshot diffing (tools exist for that, and they're noisy), but something smarter — something that understands design systems and can tell you why something is wrong, not just that it's different.

The scope felt tractable: check capitalization, verify spacing values, compare colors against our Dew Design System — Freshworks' component library with defined tokens for everything from --color-text-primary to --spacing-400.

We decided to build a Chrome extension.

Chapter 2

Token Scan — The First Approach

The initial idea was simple: scan a webpage and check if every CSS value matches a valid Dew token. No Figma required — just system compliance.

How it works

At build time, we compile all Dew Design System SCSS files into a JSON database:

103
Palette colors
114
Semantic tokens
18
Spacing values

The token database includes forward lookups (token → hex) and reverse lookups (hex → token name), so given any color on the page, we can tell you which Dew token it should be.

At runtime, a content script walks the DOM via TreeWalker, extracts every element's computed styles, and the comparison engine checks each value:

User clicks "Scan Page" ↓ Side Panel ⟶ Service Worker ⟶ Content Script ↓ getComputedStyle() on every visible element ↓ ExtractedElement[] ↓ Service Worker runs comparison-engine.ts ↓ For each color: hex ∈ token database? For each spacing: value ∈ [0,2,4,6,8,12,16,20,24,32...]? For each font: family = Inter? size ∈ type scale? ↓ VqaReport with violations

The 652-violation problem

We ran it on a Freshdesk page. 652 violations across 311 elements.

Most were noise. The page wasn't built with Dew components yet, so every color was flagged as "not a Dew token." The violations said things like "This color does not match any Dew token" — technically true, utterly useless.

We made three improvements:

  1. Human-readable element descriptions. Instead of a raw CSS selector, show "Button with 'Export' label in top-right area." We extract the tag name, text content, ARIA labels, and screen position to build plain-language descriptions.
  2. Component-aware suggestions. For Dew components (elements with data-dew), we look up the expected token from the component spec: "Secondary Button border-color should be --color-boundary-border-mildest (#eaeef2)."
  3. Closest token suggestions. For non-Dew elements, find the nearest matching token by RGB distance, filtered by property type: border-color: #c4cdd6 → "Not a Dew component. Closest match: --color-boundary-border-mild (#c9d3db)."

Token Scan works well for what it does — checking design system compliance. But it can't tell you whether the implementation matches the design. For that, we needed Figma.

Chapter 3

Figma Frame Comparison — The Big Pivot

Sivachidambaram suggested the idea that changed everything: compare the page directly against a specific Figma frame. Not just "is this a valid token?" but "does this match the design?"

The user flow:

  1. Paste a Figma frame URL (figma.com/design/...?node-id=456-789)
  2. Enter a Figma Personal Access Token (stored locally, one-time)
  3. Click "Compare with Figma"

Behind the scenes:

Figma REST API Content Script (DOM) ↓ ↓ parseFigmaUrl() getComputedStyle() fetchFigmaNodes() on every visible element flattenFigmaTree() ↓ ↓ NormalizedFigmaProps[] ExtractedElement[] ↓ ↓ ┌――――――――――┐――――――――――――――――――――――└ ↓ matchFigmaToDom() 3-pass matching ↓ comparePairByPair() ↓ FigmaComparisonReport

The matching algorithm

Figma has a tree of design nodes. The DOM has a tree of HTML elements. They look nothing alike. We needed to figure out which Figma node corresponds to which DOM element.

We built a three-pass waterfall:

PassStrategyConfidence
1Text content — match Figma TEXT nodes to DOM elements with same text1.0
2Component name — match Figma "Button" to DOM <button>0.75
3Position — normalize coordinates to [0,1], compute bounding box IoU0.3–0.9

For matched pairs, we compared properties one by one: fill color, text color, font size, font weight, border radius, padding, gap.

Result: 283 discrepancies. Most were wrong.

The matching algorithm wasn't finding the right pairs. A Figma frame would match a random DOM div, and then every property comparison was meaningless.

Chapter 4

Fighting the Noise

We did a root cause analysis of every false positive. Five problems accounted for ~95% of the noise:

CauseImpactFix
IoU threshold too low (0.3)~40%Raised to 0.5 + size similarity check
Inherited CSS values~20%Detect inherited backgrounds, skip comparison
Wrapper frames matching leaves~15%Filter FRAMEs with no visual properties
Structural nodes generating comparisons~10%Skip nodes with no fill/stroke/typography
Browser default styles~5%Skip border/font comparisons when Figma has no value

The inherited background issue was particularly insidious. getComputedStyle() returns #ffffff for every element whose parent has a white background — even if the element itself has no background set. A Figma FRAME with no fill would mismatch against every DOM element's inherited white.

The fix was a heuristic: if the element's background matches its parent's, mark it as inherited:

const bg = cs.backgroundColor
const parent = el.parentElement
if (parent) {
  const parentBg = getComputedStyle(parent).backgroundColor
  if (bg === parentBg && !el.style?.backgroundColor) {
    inheritedBg = true
  }
}

We got the discrepancies down from 283 to about 30. But then we hit a wall.

The comparisons that kept lying

Font-size, font-weight, font-family, and border comparisons produced false positives constantly. The element matching was correct by text content, but the DOM element's computed styles didn't correspond to what we expected because of CSS inheritance, specificity, and browser rendering differences.

We removed them one by one. Each time, we'd see a false positive, investigate, confirm the matching was wrong or the comparison was misleading, and disable it.

The property-by-property comparison is only as good as the element matching. And the element matching is fundamentally unreliable.

By the end, we'd stripped the comparison down to just colors and border-radius. Everything else was too noisy to trust.

Chapter 5

Content Comparison — What Actually Worked

While the visual comparison was struggling, one thing worked beautifully: text content verification.

The approach is dead simple — no element matching required:

  1. Extract every TEXT node's content from the Figma frame
  2. Collect all text from the DOM (within the scoped area)
  3. For each Figma text, check: does this exact string exist anywhere on the page?

If it doesn't, it's either missing, changed, or has a capitalization difference. No matching, no false positives from wrong element pairs.

But we discovered a subtle problem: invisible character differences.

Figma and the DOM represent text differently at the Unicode level:

CharacterFigmaDOM
Apostrophe’ U+2019' U+0027
Dash— U+2014- U+002D
Space  U+00A0 U+0020

The text looks identical to the eye but fails a strict string comparison. We built a normalization pipeline:

function normalizeTextForComparison(text: string): string {
  return text
    .replace(/[\u2018\u2019\u201A\u201B]/g, "'")  // smart quotes
    .replace(/[\u201C\u201D\u201E\u201F]/g, '"')   // double quotes
    .replace(/[\u2013\u2014]/g, '-')                // dashes
    .replace(/[\u00A0\u2000-\u200A]/g, ' ')       // special spaces
    .replace(/[\u200B\u200C\u200D\uFEFF]/g, '')    // zero-width chars
    .replace(/\s+/g, ' ').trim()
}

Even after all that, some strings still didn't match. Our final fallback: strip everything except letters and numbers and compare. If "replacethisuserspersonalinformation" matches on both sides, the text is the same.

Content comparison became the most reliable feature of the extension. It catches missing text, changed copy, and capitalization differences with zero false positives.

Chapter 6

The AI Pivot — Screenshots Over Properties

After weeks of fighting element matching, we had a realization:

Matching Figma nodes to DOM elements is a fundamentally unsolvable problem. Figma's layer structure and the DOM tree are different representations of the same visual. Trying to map between them programmatically will always be brittle.

But there's something that can compare two visuals reliably: a pair of eyes. And Claude is multimodal.

The screenshot approach

New flow, radically simpler:

User clicks "Compare with Figma" ↓ 1. chrome.tabs.captureVisibleTab() ⟶ page screenshot 2. Figma Images API ⟶ design screenshot 3. Figma node tree ⟶ property data table 4. DOM scan ⟶ CSS property data table ↓ All four sent to Claude ↓ Claude compares visually + cross-references data ↓ "font-size is 16px, should be 14px"

No element matching. No property comparison engine. No IoU thresholds. Just: here are two images and two data tables, tell me what's different.

The precision problem

The first version was imprecise. Claude would say "the label looks larger" or "typography differs." Correct, but not actionable — developers need exact values.

We tried telling Claude to use the screenshots only. That produced vague descriptions. We tried feeding it just the property data. That missed visual issues. The breakthrough was the hybrid approach:

  1. Screenshots for finding issues (what looks different?)
  2. Property tables for exact values (Figma says 14px, DOM says 16px)

But we hit another problem: Claude would look at the screenshots, identify visual issues, and then ignore the property tables when writing its report. Subtle differences (like 14px vs 16px) wouldn't be noticed visually, so they'd never be reported.

The data-first insight

We flipped the instruction order: 

// Before (missed subtle differences)
"Look at screenshots → then check data"

// After (catches everything)
"Compare the data tables first → then use screenshots for extras"

The prompt now explicitly tells Claude: 

Step 1: Compare the property tables row by row.
        Match elements by text content.
        Report EVERY value that differs.

Step 2: Check text content list against the page.

Step 3: Use screenshots for layout/visual issues
        not captured in the data.

This way, a 14px vs 16px font-size difference is caught from the data in Step 1 — Claude doesn't need to "see" it in the screenshot. The screenshots are a safety net for things the data can't express: alignment, visual weight, overall feel.

Chapter 7

What We Landed On

The extension today has two modes, each doing what it does well:

ModeWhat it checksRequires
Token ScanDew design system compliance — are CSS values valid tokens?Nothing (runs locally)
Figma CompareVisual + data + content comparison against a Figma frameFigma token + Claude API key

Figma Compare sends Claude:

  • A page screenshot (auto-captured)
  • A Figma frame screenshot (auto-fetched via API)
  • A Figma property table (font sizes, weights, colors, spacing)
  • A DOM property table (computed CSS values)
  • A Figma text content list (for content verification)

Claude returns grouped findings with exact values and copy-pasteable CSS fixes. Each finding names the specific element, states what the value is, and what it should be.

The honest assessment

Content comparison works great. It catches missing text, changed copy, and capitalization differences with near-zero false positives.

Visual comparison via AI is good but not perfect. Claude occasionally misidentifies elements or reports differences that aren't there. But the signal-to-noise ratio is dramatically better than the element-matching approach.

Element-level matching between Figma and DOM — the thing we originally set out to build — remains an unsolved problem. Figma layers and DOM elements are fundamentally different representations. The pragmatic solution is to not match them at all, and let AI do what AI is good at: looking at two pictures and telling you what's different.

The best version of this tool does fewer things than what we first imagined. But the things it does, it does correctly.

What's next

We're still iterating. Better prompt engineering could make the AI more precise. We could add Dew component spec awareness to the AI prompt, so it knows what tokens a Button should use. We could cache results so repeated comparisons are instant.

But the architecture is right: screenshots for finding issues, structured data for exact values, and AI to bridge the gap between them.

Built with TypeScript, React, esbuild, and the Chrome Extension Manifest V3 APIs. AI comparison powered by Claude (Anthropic). Figma integration via the Figma REST API.
